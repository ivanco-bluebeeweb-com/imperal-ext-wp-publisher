# WP Publisher — публикация статей из .docx в WordPress через Webbee (Imperal OS)

Сводка обсуждения от 14 июля 2026. Проект: приложение, которое читает docx-статью для блога,
разбирает её, задаёт уточняющие вопросы, учится на ответах и публикует пост в WordPress
со всеми заполненными полями.

---

## 1. Идея и принятые решения

| Вопрос | Решение |
|---|---|
| Формат инструмента | Веб-приложение (позже: Webbee интегрирована в него) |
| Действие по умолчанию | Создавать **черновик** на проверку, не публиковать сразу |
| SEO-плагин на сайте | **Rank Math** |
| Изображения | Пока без них — добавляются вручную в WP (метаданные alt/title/caption сохраняем) |
| Мультиязычность (RU/RO) | **Polylang** |
| Формат статей | **Блоки Gutenberg** |
| Платформа агента | **Imperal Cloud / Webbee** — типизированные вызовы, «чтобы ИИ не ошибался» |
| Точки входа документов | Чат с Webbee **и** «проекты» внутри веб-приложения |

## 2. Формат исходных документов

Документы сильно структурированы, что позволяет разбирать их без ИИ:

- Секции разделены линией `════`
- Метки секций: `H1:`, `LID:`, `ВЫВОД:`, `CTA:`, `SEO-БЛОК:`, `ИЗОБРАЖЕНИЯ:`
- Секция без метки — тело статьи
- SEO-блок — пары «Ключ: значение» (Meta Title, Meta Description, Slug, Основной ключ,
  Вторичные ключи, Внутренние/Внешняя ссылка, Автор, Дата, Рубрика, Формат, Язык, Зеркало)
- Изображения — подблоки `[Image N]` с полями Тип, Соотношение, Имя файла, Промт (EN), Alt, Title, Caption

**Важный нюанс:** в docx нет стилей Word (подзаголовки не Heading 2, не жирные) —
подзаголовки определяются эвристикой: короткая строка без точки в конце.

## 3. Архитектура (три слоя)

1. **Python-парсер (детерминированный)** — раскладывает документ по структуре и
   генерирует `warnings` — список того, что нужно уточнить у человека
   (незаполненная дата `2026-07-XX`, подтверждение подзаголовков).
2. **ИИ-агент (Webbee)** — задаёт вопросы только по warnings и по правилам,
   которым ещё нельзя доверять; вызывает функции расширения типизированно.
3. **База правил (обучение)** — каждое подтверждение пользователя сохраняется;
   после **3 подтверждений** правило переходит в авто-режим и вопрос исчезает.

## 4. Платформа Imperal — что выяснили из SDK (v5.9.4, PyPI)

- Расширение = класс `ChatExtension`, функции регистрируются `@chat.function`
  со строгой типизацией параметров (Pydantic; валидатор V17 запрещает `**kwargs`).
- Webbee вызывает функции напрямую, без LLM-пересказа аргументов (typed dispatch).
- `action_type="write"` включает двухшаговое подтверждение на уровне ядра платформы.
- Возможности контекста: `ctx.store` (документное хранилище — наши правила и статьи),
  `ctx.storage` (файлы — docx), `ctx.http` (запросы к WP REST), `ctx.ai.complete`,
  `ctx.config` / секреты (доступы к WP), `MockContext` для тестов.
- UI-панели: `@ext.panel(slot="center")`, компоненты `FileUpload`, `DataTable`, `Badge` и др.
- Публикация: `imperal validate` (11 «федеральных» валидаторов) → `imperal build`
  (манифест imperal.json) → `imperal deploy`; обязателен `icon.svg` (V21).

## 5. Что уже собрано

### Расширение `wp-publisher` (архив wp-publisher.zip)
- `wp_publisher/parser.py` — детерминированный разбор docx (+ разбор из bytes для ctx.storage)
- `wp_publisher/gutenberg.py` — структура статьи → разметка блоков Gutenberg
- `wp_publisher/wp_client.py` — WP REST: черновик, Rank Math meta, Polylang lang, поиск ID рубрики
- `wp_publisher/rules.py` — обучение: правила в ctx.store, авто-режим после 3 подтверждений
- `wp_publisher/main.py` — функции чата и панель «Публикации»:
  - `parse_article` (read) — docx → структура + warnings
  - `confirm_mapping` (write) — фиксация подтверждения → правило
  - `publish_draft` (write) — черновик в WP; подтверждение включает ядро платформы
  - панель: загрузка docx (FileUpload → тот же parse_article), таблица статей со статусами
  - health-check (проверка доступности WP REST), хук on_install
- Тесты: `tests/test_parser.py` — проходят (проверены на реальной статье про рекуператор)
- Статус: **`imperal validate` — 0 ошибок, 0 предупреждений; манифест собран**

### Плагин для сайта `WP Publisher Bridge` (wp-publisher-bridge-1.0.0.zip)
- Открывает REST-доступ к трём полям Rank Math
  (`rank_math_title`, `rank_math_description`, `rank_math_focus_keyword`)
- Устанавливается из админки: Плагины → Добавить новый → Загрузить плагин → zip → Активировать
- Запись полей разрешена только пользователям с правом `edit_posts`
- Есть альтернатива в виде mu-plugin (`rankmath-rest.php`) — его нельзя случайно отключить;
  плагин удобнее, но если SEO-поля вдруг пустые — первым делом проверить, активен ли он

## 6. Известные нюансы и обходы

- **Баг SDK 5.9.4:** `imperal build` пишет в манифест поля панелей (`icon`, `refresh`,
  `center_overlay`), которые схема `imperal validate` не принимает (Panel: только
  slug/tree/panel_id/title). Обход: `scripts_fix_manifest.py` — запускать после каждого build.
- **Rank Math через REST** требует плагин-мост (см. выше) — иначе meta-поля молча игнорируются.
- **Polylang:** язык передаётся параметром `?lang=ru` при создании поста; связка RU↔RO
  переводов — следующий этап, когда появятся RO-версии.
- **H1 не входит в контент** — это заголовок поста, тема выводит его сама.

## 7. Безопасность

- Application Password, случайно отправленный в чат, нужно **отозвать (Revoke)** и создать новый.
- Пароли не отправляются в чаты/письма; вводятся только в настройки расширения на панели Imperal.
